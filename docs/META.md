# Background의 Background

> 이 문서는 Sleep Thera 랜딩페이지 프로젝트를 진행하는 배경을 설명한다.

---

## 배경

- AI가 뭐든 잘 해주는 시대
- 그렇게 때문에 어느 정도 규모가 작은 프로젝트라면, 맥락을 다 아는 한 사람이 여러 AI를 쓰면 팀으로 일하는 것보다 빠를 수 있음
- 협업이 상대적인 비효율이 될 수 있다는 뜻

> 팀이 개인보다 나은 결과를 내려면, 진짜 AX(AI Transformation)가 필요

## 문제

현제 협업 플로우 예시

```
기획 도구 → [사람이 읽음] → 디자인 도구 → [사람이 읽음] → 코드(개발) → 테스트
```

- 우리는 이미 업무에서 AI를 어느 정도 쓰고 있어, 더 적극적으로 도입하기 위한 AX 플로우 구축이 필요
- 각기 다른 워크스페이스에서 각 직군의 작업이 발생한다는 현상으로 인함

| 문제 | 설명 |
|------|------|
| 일관성 확보 어려움 | 중간 단계에서 놓친 부분이 있다면 추후 재 개발 하는 참사 |
| AI에게 일 시키는 비용 | 매번 이전 단계 산출물 복붙해서 넣어줘야 함 |
| 소통 비용 | 다음 단계 담당자에게 작업 요청, 설명해주는 시간 소요 |
| AI를 충분히 못 씀 | 전체 플로우를 아는 AI가 똑똑한데 이전 단계의 산출물을 충분히 활용 못함 (제품 퀄리티에 영향) |
| 산출물 별 버전 싱크 안 됨 | "그게 몇월 버전이였죠?" — 형상관리 비용 높음(소통 및 규제 문서 작성 effort가 들 수 밖에 없음)|
| 팀워크 위험 | "나 혼자 하는 게 더 빠른데" 각자의 전문성을 발휘하지 못하는 마찰|

---

## 목표

> 팀원들이 생각하고 검수하는 일에 집중할 수 있도록 AI를 활용하는 과정에서 휴먼 리소스는 최소화하고 AI에 입력되는 정보의 질은 높인다.

---

## 요구사항

1. **같은 버전을 싱크** — 기획, 디자인, 개발이 항상 같은 버전을 바라본다
2. **이전 단계 산출물 활용** — 각 단계에서 이전 산출물을 AI에게 바로 넘길 수 있어야 한다

이를 위해 Git 중심으로 일하고, 기획 문서를 Markdown으로 작성한다. 모든 산출물이 같은 저장소에 있으면 버전 싱크가 자동으로 해결되고, AI가 이전 단계 산출물을 바로 참조할 수 있다.

### 특히 왜 Git 중심으로 일하는가
의료기기는 형상 관리가 필수이며, 변경 이력의 추적을 요구한다
- "누가, 언제, 왜 변경했는가"를 증명할 수 있어야 한다
- 특정 버전의 제품을 언제든 다시 빌드할 수 있어야 한다

Git을 중심에 두면 버전 싱크와 형상 관리를 동시에 해결할 수 있다.

| 산출물 | Git 기반 관리 |
|--------|---------------|
| 기획 문서 | GitHub Markdown, PR로 변경 관리 |
| 디자인 명세 | GitHub Markdown (TBD) |
| 코드 | GitHub, 커밋 메시지에 의도 기록 |

모든 산출물이 같은 버전 히스토리를 공유하면 제품 개발 속도도 높아지며, 각 산출물의 시점을 역추적하는 문제가 해결된다. (형상 관리가 업무 효율을 높여주는 방향으로 작동)

## How

각 프로세스에 AI를 붙이려면 기존 방식에서 전환이 필요

- **기획**: 암묵적 맥락 → 명시적 문서화 (AI가 읽을 수 있도록)
- **디자인**: 명세 문서 → AI 생성 → 검토/수정
- **개발**: 명세 문서 기반 AI 코드 생성 → 리뷰/수정
- 암묵지 → 형식지: 머릿속에 있는 것을 텍스트로 꺼내야 AI가 활용 가능
- 결과물 중심 → 의도 중심: "이렇게 만들어줘"가 아니라 "이런 목적이야"를 포함
> 각 단계를 원활하게 잇는 구조가 필요

### Learning by Doing

1. 직접 겪어보며 필요한 것을 발견한다
2. 도구가 없으면 직접 만든다
3. 완벽한 설계 없이 하면서 개선한다

- 우리 팀에 맞는 AI 협업 워크플로우는 아직 Best Practice가 없는 새로운 영역. 직접 해보며 만들어야 함.

### 유지보수보다 새 피처/프로젝트 런칭에 초점을 맞춤
- 레거시 혼재 방지: 아직 정리되지 않은 기존 프로젝트에 새 방식이 섞이면 오히려 복잡해짐
- 실패 격리: 실험 실패해도 기존 시스템에 영향 없음
- 빠른 검증: 작은 범위에서 빠르게 돌려보고 판단
- 워크플로 변경 리스크 격리: 이론적으로 필요한 것과 실제로 필요한 것이 다를 수 있다. 미리 설계하면 실제로 막히는 지점을 놓친다. 기존 프로젝트에 붙였다가 더 나은 방식을 발견하면 다시 뜯어고쳐야 함

### 랜딩페이지
새 프로젝트로 Sleep Thera 랜딩 페이지를 만들 예정

- 아직 우리에게 없지만 실제로 필요한 것: 지금 테스트 유저 풀 확보가 필요 (BACKGROUND.md 참조)
- 팀 전문성 발휘: 워크플로우 구축을 위한 기획, 디자인, 프론트엔드, 백엔드 모든 역할이 참여하면서 너무 무겁지 않은 스펙

## 프로벡트 성공 기준

### 1차 지표: 효율 측정

**측정 항목**
| 항목 | 설명 |
|------|------|
| Estimation | 티켓 작업 소요 시간: 이전 프로세스 예상 소요시간 대비 실제 소요 시간 |
| 맥락 전달 비용 | 업무 전달 과정에서 소통에 든 노력 |
| AI 활용도 | AI에게 맥락 전달하는 수고, 결과물 활용 정도 |


**측정 방법**
프로젝트 완료 후 팀 설문. 각 항목에 대해 두 가지 응답:
- A: 이번 프로젝트
- B: 기존 방식으로 했으면 어땠을지 (상상)

**판단**: A < B (새 방식이 더 낫다고 체감)

### 2차 지표: 체크리스트

| 항목 | 확인 |
|------|------|
| 배포 완료 | 실제 URL 접속 가능 |
| 기획 문서 | GitHub Markdown으로 작성됨 |
| 워크플로우 도식화 | 어떤 작업을 어디서(도구) 누가(인간/AI) 하는지, 단계별 산출물 흐름을 도식화 |

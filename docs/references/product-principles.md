# 슬립테라 제품 원칙 (Product Principles)

> **미션**: "슬립테라는 사용자가 밤을 두려워하지 않게 만듭니다."

---

## 개요

이 문서는 슬립테라의 모든 제품 결정, 디자인, 커뮤니케이션의 기준이 되는 핵심 원칙을 정의합니다. HCI(Human-Computer Interaction) 관점에서 사용자와의 관계, 동기 설계, 신뢰 구축의 세 축으로 구성됩니다.

---

## 1. 도우미 (Helper) — 관계적 역할

### 정의
슬립테라는 사용자에게 "도구"가 아닌 "파트너"입니다. 수면 여정을 함께하는 동행자로서 관계를 맺습니다.

### 인터랙션 원칙

| 원칙 | 설명 | 예시 |
|------|------|------|
| **대화형 협력** | 일방적 명령-실행이 아닌 쌍방향 소통 | "오늘 밤 목표를 같이 정해볼까요?" |
| **맥락 인식** | 사용자의 상황(컨디션, 수면 패턴)을 기억하고 반영 | "어제 늦게 주무셨네요. 오늘은 좀 일찍 시도해볼까요?" |
| **비판단적 어조** | "해라"가 아닌 "같이 해보자" | "괜찮아요, 천천히 가봐요" |

### 경험 목표
- 혼자가 아니라는 느낌
- 판단받지 않는 안전한 공간
- 장기적 동행자로서의 신뢰감

### 디자인 체크리스트
- [ ] 실패/이탈 시 비난하지 않는 피드백 제공
- [ ] 사용자 이름 및 상황 맥락 자연스럽게 활용
- [ ] 명령형 문장 대신 제안형 문장 사용

---

## 2. 내적 만족감 (Intrinsic Satisfaction) — 동기 설계

### 정의
Self-Determination Theory에 기반하여, 외적 보상(포인트, 뱃지, 랭킹)이 아닌 내적 동기를 유발합니다.

### 내적 동기의 세 요소

| 요소 | 정의 | 슬립테라 적용 |
|------|------|--------------|
| **자율성 (Autonomy)** | 스스로 선택하고 통제하는 느낌 | 사용자가 목표와 방법을 직접 선택 |
| **유능감 (Competence)** | 성장하고 있다는 느낌 | 작은 성공 경험의 축적과 인정 |
| **관계성 (Relatedness)** | 연결되어 있다는 느낌 | 에이전트와의 정서적 유대 |

### 인터랙션 원칙

| 원칙 | 설명 | 예시 |
|------|------|------|
| **부드러운 제안** | 강제 알림 대신 선택권 제공 | "준비되면 알려주세요" |
| **의미 부여** | 숫자가 아닌 의미 중심 피드백 | "오늘 10분 더 일찍 잤네요. 작은 변화가 쌓이고 있어요" |
| **과정 인정** | 결과보다 노력에 대한 인정 | "시도한 것만으로도 충분해요" |

### 경험 목표
- "앱 때문에"가 아니라 "내가 해냈다"는 성취감
- 지속 사용의 이유가 보상이 아닌 의미
- 자기 효능감(self-efficacy)의 향상

### 디자인 체크리스트
- [ ] 게이미피케이션 요소(뱃지, 랭킹, 스트릭) 최소화
- [ ] 리플렉션 유도 질문 포함 ("오늘 밤은 어땠어요?")
- [ ] 수치 중심이 아닌 서사 중심의 피드백 제공
- [ ] 실패를 "다시 시작할 기회"로 프레이밍

---

## 3. 신뢰 (Reliable) — 시스템 신뢰성

### 정의
건강 도메인에서 필수적인 신뢰를 세 층위로 구축합니다. 검증된 기술과 임상 결과를 바탕으로 한 신뢰 경험을 전달합니다.

### 신뢰의 세 층위

| 층위 | 질문 | 슬립테라 적용 |
|------|------|--------------|
| **기능적 신뢰** | "제대로 작동하는가?" | 정확한 수면 분석, 버그 없는 안정적 경험 |
| **정보적 신뢰** | "정보가 정확한가?" | CBT-I 기반, 임상 근거 명시, 출처 제공 |
| **정서적 신뢰** | "나를 위하는가?" | 상업적 의도 최소화, 프라이버시 존중 |

### 인터랙션 원칙

| 원칙 | 설명 | 예시 |
|------|------|------|
| **솔직한 불확실성** | 모르는 것은 모른다고 인정 | "이 부분은 확실하지 않아요" |
| **근거 제시** | 정보의 출처와 배경 설명 | "이 방법은 OO 연구에 기반해요" |
| **투명한 설명** | 데이터 사용 목적 명확히 전달 | "이 질문은 OO를 파악하기 위해 드려요" |

### 경험 목표
- "이 앱은 나를 팔아먹지 않는다"는 안심
- "여기서 하는 말은 근거가 있다"는 확신
- 의료 앱 수준의 진지함과 책임감

### 디자인 체크리스트
- [ ] 온보딩에서 CBT-I 배경 및 효과 설명 포함
- [ ] "왜 이걸 물어보나요?" 설명 제공 (툴팁 또는 링크)
- [ ] 민감 정보 수집 시 목적과 사용 범위 명시
- [ ] 개인정보 처리 방침 쉽게 접근 가능하도록 배치

## 통합 프레임워크

```
┌─────────────────────────────────────────────────┐
│                사용자 경험 목표                    │
│       "밤이 두렵지 않은 나"로의 변화                │
└─────────────────────────────────────────────────┘
                        ▲
          ┌─────────────┼─────────────┐
          │             │             │
     ┌────┴────┐  ┌─────┴─────┐  ┌────┴────┐
     │ Helper  │  │ Intrinsic │  │Reliable │
     │ (관계)   │  │  (동기)    │  │ (신뢰)   │
     └────┬────┘  └─────┬─────┘  └────┬────┘
          │             │             │
     동행하는        의미 있는       검증된
     파트너          성장 경험      안전한 기반
```

### 원칙 간 상호작용
- **Helper + Intrinsic**: 파트너 관계가 내적 동기의 "관계성" 요소를 강화
- **Intrinsic + Reliable**: 신뢰가 있어야 의미 있는 경험이 가능
- **Reliable + Helper**: 신뢰할 수 있어야 진정한 파트너가 될 수 있음

## 적용 가이드

### 카피라이팅 시
1. 명령형 → 제안형으로 변환
2. 숫자 중심 → 의미 중심으로 변환
3. 근거와 출처 자연스럽게 포함

### UI/UX 설계 시
1. 강제 요소 최소화, 선택권 부여
2. 맥락 인식 기능 우선 고려
3. 실패 케이스에 대한 따뜻한 대응 설계

### 기능 기획 시
1. "이 기능이 세 원칙 중 어디에 기여하는가?" 질문
2. 외적 보상보다 내적 의미 창출에 집중
3. 신뢰 저해 요소(과도한 데이터 수집, 불투명한 알고리즘) 제거
